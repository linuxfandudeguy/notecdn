Why You Shouldn't Use SRI with Dynamically Generated Files

Subresource Integrity (SRI) is a security feature that allows browsers to verify that files they fetch (such as scripts and stylesheets) are delivered without unexpected manipulation. However, SRI is not suitable for dynamically generated files due to the following reasons:

Key Reasons:

  1. Hash Mismatch: SRI relies on a fixed hash value specified in the HTML to verify the integrity of a file. When a file is dynamically generated, its content can change with each request or update, resulting in a different hash value. This causes the SRI check to fail, and the browser may block the resource.
  2. Practicality: Managing SRI hashes for dynamically generated files is impractical because updating the hash every time the content changes is not feasible. This undermines the purpose of using SRI for ensuring the integrity of fetched resources.

Example of What Not to Do

Do not use SRI with dynamically generated files:

<script src="https://notecdn.vercel.app/fetch/npm/axios/dist/axios.min.js" integrity="sha256-abc123456789abcdef..." crossorigin="anonymous"></script>

In this example, axios.min.js is dynamically generated and changes its content frequently, causing the integrity hash sha256-abc123456789abcdef... to become outdated quickly.

What to Do Instead

For dynamically generated files, consider alternative security practices:

<script src="https://notecdn.vercel.app/fetch/npm/axios/dist/axios.min.js" crossorigin="anonymous"></script>

Ensure that the server delivering the script uses strong security measures to prevent tampering. Regularly audit and monitor the server's integrity and implement other security mechanisms such as Content Security Policy (CSP) to enhance security.

Conclusion

For resources that are dynamically generated or frequently updated, it's recommended to use security practices other than SRI to ensure their integrity and security.

For more information on using SRI effectively, visit MDN Web Docs - Subresource Integrity.
